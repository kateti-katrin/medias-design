# СМЫСЛ — шпаргалка к защите проекта

## 1. Как сказать о проекте за 20 секунд

Это медиа-проект о дизайне для заказчиков.
В проекте есть главная страница, страница со статьями, 4 отдельные статьи, 2 сервиса: генератор ТЗ и калькулятор бюджета.
Технически это Rails-приложение: страницы рендерятся на сервере, стили собраны через SCSS, интерактив вынесен в JavaScript-модули.
На backend есть статьи, теги, лайки, комментарии, просмотры, авторизация через Devise и API-контур для JSON-запросов.

## 2. Две версии проекта

- GitHub Pages — статическая витрина проекта. Нужна, чтобы быстро показать дизайн, страницы, статьи, сервисы и 404.
- localhost:3000 — живая Rails-версия. Нужна, чтобы показать backend, API, базу, Devise, просмотры, лайки и JSON-маршруты.

Коротко: GitHub Pages показывает интерфейс, а Rails показывает полную логику.

## 3. Как устроен проект

- routes — адреса страниц и API.
- controllers — решают, что отдать пользователю.
- models — описывают данные и связи.
- views — сами экраны.
- SCSS — внешний вид.
- JavaScript — интерактив.

Главные файлы:

- config/routes.rb — все маршруты.
- app/controllers/articles_controller.rb — обычные страницы статей.
- app/controllers/api/articles_controller.rb — JSON для статей.
- app/models/article.rb — главная модель контента.
- app/models/tag.rb — модель тегов.
- app/models/like.rb — лайки.
- app/models/comment.rb — комментарии и ответы.
- app/models/user.rb — Devise-пользователь.
- app/models/profile.rb — профиль пользователя.
- app/views/pages/home.html.erb — главная.
- app/views/articles/index.html.erb — страница «Читать».
- app/views/articles/show.html.erb — отдельная статья.
- app/javascript/application.js — подключение интерактива.
- app/assets/stylesheets/application.scss — сборка стилей.

## 4. Как работает страница статьи

Пример сценария:

1. Пользователь открывает URL статьи.
2. Маршрут отправляет запрос в ArticlesController.
3. Контроллер находит статью по slug.
4. Модель Article хранит заголовок, текст, тег, картинку, просмотры, лайки и комментарии.
5. View рисует страницу статьи.
6. JavaScript после загрузки отправляет запрос на просмотры и получает состояние лайка.

Короткая формула:
route -> controller -> model -> view -> JS.

## 5. Где API и что он делает

Основные endpoints:

- GET /api/ping — проверка, что API отвечает.
- GET /api/articles — список статей в JSON.
- GET /api/articles/:id — одна статья в JSON.
- GET /api/articles/:article_id/like — текущее состояние лайка.
- POST /api/articles/:article_id/like — поставить лайк.
- DELETE /api/articles/:article_id/like — убрать лайк.
- GET /api/articles/:article_id/comments — список комментариев.
- POST /api/articles/:article_id/comments — создать комментарий.
- POST /api/login — вход по JSON.
- DELETE /api/logout — выход по JSON.
- POST /articles/:id/track_view — увеличить просмотры статьи.

Очень простое объяснение:
API — это способ получать не HTML-страницу, а данные в JSON.

## 6. Главные модели простыми словами

### Article
Главная контентная модель проекта.
Хранит title, body, excerpt, slug, public_uuid, cover image, views_count и связи с тегом, лайками и комментариями.

### Tag
Отдельная модель рубрик.
Нужна, чтобы теги были не просто подписями на карточках, а реальными данными для фильтрации.

### Like
Лайк посетителя.
Сделан как polymorphic association: в будущем такой механизм можно использовать не только для статей.

### Comment
Комментарий к статье.
Есть self join: комментарий может быть ответом на другой комментарий.

### User и Profile
User отвечает за вход и регистрацию через Devise.
Profile хранит display_name, bio и avatar_url.

## 7. Какие Rails-паттерны реализованы

- I18n — русский язык по умолчанию.
- Devise — регистрация, вход, выход.
- slug URL — красивые адреса для статей.
- SEO/meta/OG — title, description, open graph, canonical.
- CarrierWave — обложка статьи через backend.
- views counter — просмотры статьи через запрос к backend.
- polymorphic association — Like.
- self join — Comment -> replies.
- has_one — User -> Profile.
- nested form — редактирование profile внутри user.
- STI — несколько типов статей в одной таблице.
- UUID — внешний публичный идентификатор статьи.
- JWT — JSON-аутентификация для API.
- CORS — разрешение внешних API-запросов.

## 8. Как объяснить сервисы

### Генератор ТЗ
Пользователь отвечает на 8 вопросов.
JavaScript собирает ответы и формирует черновик технического задания, который можно скопировать.

### Калькулятор бюджета
Пользователь выбирает тип проекта, количество страниц, доп. опции и срочность.
Сервис считает ориентировочный диапазон стоимости и сроков.

## 9. Как отвечать на частые вопросы преподавателя

### Почему это Rails-проект, а не просто набор HTML?
Потому что здесь есть маршруты, контроллеры, модели, база, Devise, API, slug, лайки, комментарии и просмотры.

### Почему статьи открываются по красивому адресу?
Потому что у статьи есть slug, и он используется в URL вместо обычного id.

### Зачем отдельная модель Tag?
Чтобы теги были частью backend-структуры, а не просто текстом на карточке.

### Что такое polymorphic association?
Это гибкая связь, когда одна модель Like может в будущем работать с разными сущностями, а не только со статьёй.

### Что такое self join?
Это связь модели с самой собой: комментарий может иметь родительский комментарий и ответы.

### Зачем JWT, если уже есть Devise?
Devise нужен для обычной браузерной авторизации, а JWT — для API-сценария с JSON.

### Почему есть GitHub Pages и localhost отдельно?
GitHub Pages показывает дизайн как витрину, а Rails локально показывает полную backend-логику.
